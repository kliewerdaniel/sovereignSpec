# ADR-006: SQLite as Primary Metadata Store

**Status:** Accepted

**Date:** 2025-01-01

## Context

SovereignSpec requires persistent metadata storage for:
- Project records
- Specification records (with versioning, checksums, and status tracking)
- Spec relationships (edges in the knowledge graph, stored for persistence beyond graph.json)
- ADR records
- Task records (with status, agent assignment, and output tracking)
- Agent session records
- Artifact records (file paths, validation status, checksums)
- Pattern records
- Session history

The storage system must be:
- **Local-first**: No server process, no cloud database, no network connection
- **Zero configuration**: Must work immediately after `sovereignspec init` with no setup
- **Relational**: The data is inherently relational (specs have dependencies, tasks belong to specs, artifacts belong to tasks)
- **Transactional**: Spec compilation involves multiple writes that must be atomic
- **Portable**: The database file should be movable between machines
- **Lightweight**: Should not add significant startup time or memory overhead

## Decision

SovereignSpec will use SQLite as its primary metadata store, accessed via Python's built-in `sqlite3` module (with optional use of Drizzle ORM for the Next.js UI layer).

The SQLite database lives at `.sovereignspec/memory/sovereignspec.db`. It contains 10 tables: `projects`, `specifications`, `spec_relationships`, `spec_versions`, `adrs`, `tasks`, `agents`, `artifacts`, `patterns`, and `sessions`.

The schema uses TEXT for UUIDs, TEXT for ISO 8601 timestamps, and JSON text columns for flexible metadata (`metadata_json`, `contradictions_json`, `capabilities_json`). Indexes are created on foreign keys and frequently-queried columns (status, spec_id, relationship_type).

## Rationale

SQLite is the right choice for this use case for the following reasons:

1. **Bundled with Python**: The `sqlite3` module is part of Python's standard library. No additional dependencies are needed for core functionality.
2. **Zero configuration**: SQLite requires no server process, no configuration files, no daemon management. The database file is created on first access.
3. **Transactional**: SQLite supports ACID transactions with `BEGIN`/`COMMIT`/`ROLLBACK`. This is essential for the spec compiler pipeline, which must atomically update the spec record, create relationships, add version history, and register contradictions.
4. **Portable**: A single `.db` file contains the entire metadata store. It can be copied, backed up, or moved between machines.
5. **Relational**: The metadata model is inherently relational. SQLite's SQL support (including recursive CTEs for graph queries) is sufficient for all query needs.
6. **Performance**: SQLite handles single-user workloads (the typical SovereignSpec use case) with excellent performance. The entire database is typically < 10MB for a project with hundreds of specs.

## Alternatives Considered

| Alternative | Reason for Rejection |
|-------------|---------------------|
| PostgreSQL | Requires server installation, configuration, and connection management. Overkill for local-first single-user. |
| MySQL | Same issues as PostgreSQL, plus heavier footprint. |
| MongoDB | Document store not ideal for inherently relational data. No joins. More complex setup. |
| DuckDB | Excellent for analytics but not designed for transactional workloads. No concurrent write support. |
| JSON files (only) | Lacks queries, indexes, transactions, and integrity constraints. |
| SQLite via Drizzle ORM | Viable for the UI layer but adds a dependency for the Python core. |
| LMDB | Key-value store; requires manual index management. No SQL queries. |

## Consequences

### Positive

- Zero external dependencies for the metadata store (batteries included in Python)
- ACID transactions for spec compilation pipeline
- SQL queries for complex reporting (e.g., "find all specs with drift_score < 0.6 that were updated in the last week")
- Single file for backups and migrations
- Fast startup (no connection pooling, no authentication, no network)
- Well-understood concurrency model (WAL mode for reads, exclusive locks for writes)

### Negative

- SQLite does not support concurrent writers (not an issue for single-user mode but could be if multiple agents write simultaneously)
- No built-in user authentication or access control (not needed for local-first but limits multi-user deployment)
- Schema migrations require manual management (versioned SQL files in `persistence/migrations/`)
- SQLite has limited ALTER TABLE support (can't drop columns or modify constraints without recreating tables)
- Binary data (embeddings) is stored in ChromaDB, not SQLite — split storage introduces synchronization concerns

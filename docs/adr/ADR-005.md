# ADR-005: Agent Agnostic Bootstrap Protocol

**Status:** Accepted

**Date:** 2025-01-01

## Context

SovereignSpec must integrate with a wide range of AI coding agents: Claude Code, OpenCode, Cursor, Cline, RooCode, Codex CLI, Gemini CLI, Aider, Windsurf, Continue, and any future agents that emerge. Each agent has its own integration conventions:
- Agent-specific instruction files (CLAUDE.md, AGENTS.md, GEMINI.md)
- Custom rules directories (.claude/commands/, .cursor/rules/, .continue/commands/)
- Configuration files (.aider.conf.yml, .continue/config.json)
- Project-level rule files (.clinerules, .windsurfrules, .roo/rules.md)

Building and maintaining a custom plugin for each agent is unsustainable. Each agent has a different plugin API, a different release cycle, and a different set of capabilities. By the time SovereignSpec v1.0 ships, at least three more coding agents will likely exist.

The central question: should SovereignSpec build agent-specific plugins, or should it communicate with all agents through a single universal mechanism?

## Decision

SovereignSpec will communicate with all coding agents exclusively through the filesystem. There will be no API integration, no plugin code, no custom protocol for any agent. The integration mechanism is:

1. **`.sovereignspec/bootstrap.md`**: A universal contract file that any file-aware coding agent can read to understand the project. This file contains the full agent contract, spec format reference, task format, artifact submission protocol, and constraints.

2. **Agent-specific integration files**: SovereignSpec's `integrate` command writes the appropriate instruction file(s) for each agent (CLAUDE.md, AGENTS.md, .clinerules, etc.), but each of these files simply points to `.sovereignspec/bootstrap.md` as the authoritative source.

3. **Filesystem as API**: Agents write artifacts to `.sovereignspec/agents/{name}/artifacts.json`. SovereignSpec reads these files. Agents read specs from `.sovereignspec/specs/`. SovereignSpec writes them. This bidirectional filesystem communication is the entire integration surface.

4. **No custom plugins**: SovereignSpec will never create a Claude Code plugin, an OpenCode plugin, or any agent-specific plugin. The bootstrap.md + integrate command pattern is the complete integration strategy.

## Rationale

The filesystem-as-API approach has three decisive advantages over plugin-based integration:

1. **Zero maintenance**: When an agent changes its API, plugin integrations break and must be updated. Filesystem integration does not break because files are the universal primitive that every agent supports. The `.sovereignspec/bootstrap.md` file does not change format when an agent updates.

2. **Future-proof**: A new coding agent can integrate with SovereignSpec by simply being file-aware. No SovereignSpec code changes are needed. The new agent's integration is limited to writing one file (its instruction file convention) that points to bootstrap.md.

3. **Universality**: Every coding agent can read files. Every coding agent can write files. This is the only integration mechanism that works for all agents without exception. Plugin APIs are agent-specific — filesystem access is universal.

The `bootstrap.md` file is designed to be self-contained and unambiguous. An agent that has never heard of SovereignSpec can read this file and understand exactly what it needs to do. No prior knowledge, no training, no configuration.

## Alternatives Considered

| Alternative | Reason for Rejection |
|-------------|---------------------|
| Agent-specific plugins | Breaks when agent APIs change; requires per-agent development |
| MCP (Model Context Protocol) | Promising but not yet universal; not all agents support it |
| HTTP API integration | Requires agents to make HTTP calls; not all agents support this |
| Custom agent SDK | Monolithic; couples SovereignSpec to specific agent implementations |
| VS Code extension | Only works in VS Code; not all agents run in VS Code |
| Git hooks only | Too limited; cannot communicate task context or artifact requirements |

## Consequences

### Positive

- Zero agent-specific code in the SovereignSpec engine core
- New coding agents integrate by writing one file — no SovereignSpec changes needed
- The bootstrap.md file is a standalone contract that documents the entire integration surface
- Agent updates never break SovereignSpec integration
- Works with any file-aware coding agent, including those not yet invented

### Negative

- Filesystem communication is slower than direct API calls (file read/write overhead)
- The bootstrap.md file is large (~200 lines) because it must be self-contained
- Some agents may not read `.sovereignspec/bootstrap.md` unless explicitly pointed to it
- Artifact submission requires agents to write well-formed JSON (potential for malformed submissions)
- No real-time streaming feedback from agents to SovereignSpec

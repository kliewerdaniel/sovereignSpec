# ADR-003: GBNF Grammar Enforcement

**Status:** Accepted

**Date:** 2025-01-01

## Context

Local LLMs without output constraints exhibit significant variability in generated output. When prompted to produce a JSON structure, they frequently:
- Omit required fields (e.g., omitting `contradiction_score` from a contradiction report)
- Add fields not in the schema (e.g., adding an `explanation` paragraph alongside structured data)
- Hallucinate invalid values (e.g., producing a score of "likely" instead of a float between 0 and 1)
- Mix natural language with structured output (e.g., "Here is the result: {json}")
- Produce syntactically invalid JSON (missing commas, unescaped quotes, trailing commas)

This variability is unacceptable for deterministic spec-driven development. If compilation output varies between runs, the system cannot guarantee reproducible implementations.

Existing approaches to constraint include:
1. **Post-hoc parsing and retry**: Generate free-form text, try to parse it, retry on failure. This is slow, expensive (N+1 LLM calls), and unreliable (the LLM may repeat the same error).
2. **Few-shot prompting**: Include examples of valid output in the prompt. This works for large models but degrades significantly for smaller local models (7B-32B parameters).
3. **Output format instructions**: Adding "Output JSON only" to the prompt. This is surprisingly unreliable — even large models frequently add commentary.

## Decision

SovereignSpec will use GBNF (GGML BNF) grammar files to constrain the token probability space of the local LLM during generation. Every structured output type — contradiction scores, implementation plans, task lists, API specs, ADRs, test cases, contradiction reports, and drift reports — has a corresponding GBNF grammar file in `.sovereignspec/grammar/`.

The grammar is passed to Ollama's `/api/generate` endpoint via the `options.grammar` field. The LLM's token sampler is restricted to only tokens that match the grammar's rules at each step. This is not a post-hoc filter — it is a generation-time constraint.

## Rationale

GBNF grammar enforcement provides three guarantees that no other approach can match:

1. **Structural validity**: The output is guaranteed to match the grammar's schema. If the grammar defines a JSON object with required fields `contradiction_score` (float) and `description` (string), the LLM cannot generate anything else.
2. **Determinism at fixed temperature**: With temperature=0 and a grammar constraint, the same prompt always produces the same output. This is essential for reproducible compilation.
3. **Speed**: Grammar-constrained generation is not slower than unconstrained generation — the grammar prunes invalid tokens from the probability distribution, which can actually speed up sampling.

GBNF was chosen over alternatives because it is natively supported by llama-cpp (Ollama's backend), has no external dependencies, and is a straightforward BNF-like syntax that is easy to write and debug.

## Alternatives Considered

| Alternative | Reason for Rejection |
|-------------|---------------------|
| Post-hoc JSON parsing with retry | Unreliable; N+1 LLM calls; no determinism guarantee |
| Few-shot prompting | Model-dependent; fails with small models |
| JSON mode (Ollama `format: json`) | Only constrains top-level JSON structure, not field types or values |
| Guidance (Microsoft) | External dependency; not natively supported by Ollama |
| Outlines (dot-agent) | External dependency; requires integration layer |
| LMQL | External dependency; overkill for this use case |

## Consequences

### Positive

- Deterministic output: same prompt + same grammar = same result (at fixed temperature)
- No post-hoc parsing errors: all generated output is structurally valid by construction
- No hallucinated fields: the grammar defines exactly what can appear
- Well-documented schema: the grammars serve as living documentation of each output type
- Simple debugging: a malformed grammar produces obvious errors (no tokens match)

### Negative

- Each output type requires a separate grammar file (8 files for v1)
- Complex schemas result in large grammar files that are verbose to write
- Grammar errors can be cryptic (no tokens match the grammar → generation fails)
- Not all Ollama models support grammars equally well (some have bugs with complex grammars)
- Grammar-constrained generation may produce lower-quality output for creative tasks (but all SovereignSpec outputs are structured, not creative)

# ADR-004: ChromaDB for Vector Storage

**Status:** Accepted

**Date:** 2025-01-01

## Context

SovereignSpec requires semantic search over specifications to enable:
- **RAG-grounded clarification**: When a user asks to clarify a spec, the system must retrieve related specs from vector similarity.
- **Contradiction detection**: Related specs must be identified by semantic similarity before LLM analysis.
- **Repository pattern matching**: When generating code, the system should retrieve relevant coding patterns by semantic similarity to the spec's requirements.

The core requirement is a local-first vector database that runs entirely on the user's machine with no cloud dependency. The vector store must support:
- Cosine similarity search
- Metadata filtering
- Persistence to disk
- Embedding computation via local Ollama models

## Decision

SovereignSpec will use ChromaDB as its vector store, running in embedded mode within the same Python process. Configuration:

- **Collections**: Three collections — `sovereignspec_specs` (spec content embeddings), `sovereignspec_adrs` (ADR content embeddings), `sovereignspec_patterns` (pattern description embeddings)
- **Embedding model**: `nomic-embed-text` via Ollama (default), with `all-MiniLM-L6-v2` via sentence-transformers as fallback
- **Distance metric**: Cosine similarity (default), L2 as fallback
- **Persistence path**: `.sovereignspec/memory/chromadb/`
- **Chunking**: Specs chunked by section with 512-token overlap

## Rationale

ChromaDB was chosen because it is the only mature local-first vector database that:
1. Runs embedded in-process (no server process required)
2. Has a simple Python API compatible with the existing tech stack
3. Supports multiple distance metrics including cosine similarity
4. Persists to a configurable local path
5. Has no external dependencies beyond the Python package

ChromaDB's "embedded mode" aligns perfectly with SovereignSpec's local-first architecture: the vector store starts and stops with the SovereignSpec process, requires no configuration, and stores data in a predictable path within the project directory.

## Alternatives Considered

| Alternative | Reason for Rejection |
|-------------|---------------------|
| Qdrant | Requires running a separate server; overkill for single-user |
| Pinecone | Cloud-only; violates local-first principle |
| Weaviate | Requires Docker; too heavy for a local-first tool |
| FAISS | Excellent for search but no built-in persistence, metadata filtering, or collection management |
| LanceDB | Good alternative but newer ecosystem; ChromaDB has broader community adoption |
| SQLite + FTS5 | Full-text search is not semantic search; cannot match "authentication" with "login" |
| Annoy (Spotify) | No persistence or metadata filtering; read-only index after build |

## Consequences

### Positive

- Fully embedded: no server process, no configuration, no cloud dependency
- Simple API: `chroma_client.get_collection().query(query_embeddings=...)` 
- Good community support and documentation
- Persistence to project directory means the vector store is part of the project (can be git-ignored or included)
- Supports metadata filtering for spec status, type, and version queries

### Negative

- ChromaDB's Python embedding function API is somewhat convoluted (requires subclassing)
- Large collections (>10,000 embeddings) may have slower query times due to lack of indexing
- No built-in support for hybrid search (vector + keyword)
- ChromaDB's collection management has rough edges (deleting collections sometimes leaves orphaned data)
- The embedded mode means all embeddings are loaded into memory on startup

# ADR-002: Specification Graph Node Model

**Status:** Accepted

**Date:** 2025-01-01

## Context

Traditional Spec-Driven Development tools store specifications as flat markdown files. While human-readable, flat files have fundamental limitations:

1. **No relationship tracking**: A spec that "depends on" another spec is just text — the tool cannot traverse dependency chains.
2. **No impact analysis**: There is no way to answer "what breaks if this spec changes?" without reading every spec manually.
3. **No cross-spec contradiction detection**: Contradictions between specs are invisible until implementation reveals them.
4. **No architectural traceability**: The relationship between specs, ADRs, tasks, and code modules is implicit and untracked.
5. **No drift measurement**: The system cannot compute how much a spec has drifted from the project constitution because there is no structured representation of spec content.

The central question was: should SovereignSpec model specs as documents (files with content) or as nodes (entries in a graph with typed relationships)?

## Decision

Specifications will be modeled as nodes in a directed knowledge graph, not as flat documents. The graph has 11 node types (Project, Feature, Specification, Module, Service, Endpoint, Database, ADR, Task, Agent, Document) and 9 relationship types (IMPLEMENTS, DEPENDS_ON, REFERENCES, GENERATES, REPLACES, SUPERSEDES, CONFLICTS_WITH, RELATED_TO, VALIDATES).

The `.sspec` file is still the human-authored artifact, but it is compiled into a graph node. The graph is the runtime representation; the `.sspec` file is the serialization format.

The default graph storage is adjacency JSON (`.sovereignspec/graph/graph.json`), requiring zero external dependencies. Neo4j is available as an optional upgrade.

## Rationale

A graph model enables:

1. **Impact analysis queries**: `what_breaks_if_changed(spec_id)` traverses DEPENDS_ON edges to find all affected nodes. This is a graph traversal, not a full-text search.
2. **Dependency resolution**: Topological sort of DEPENDS_ON edges determines compilation and implementation ordering.
3. **Contradiction detection**: CONFLICTS_WITH edges connect contradictory specs. The LLM creates these edges during analysis, and the graph stores them permanently.
4. **Architecture traceability**: REFERENCES edges from specs to ADRs answer "which ADR created this pattern?"
5. **Narrative drift**: By embedding spec nodes and comparing against the constitution vector (also a graph attribute), drift becomes a quantifiable metric.

The adjacency JSON format was chosen for default storage because it is a well-known, dependency-free format that is human-readable, diffable in git, and trivially parseable in any language. Neo4j provides a path for larger projects that need transactional graph operations or Cypher queries.

## Alternatives Considered

| Alternative | Reason for Rejection |
|-------------|---------------------|
| Flat markdown only (Spec Kit approach) | No relationship tracking, no impact analysis |
| SQL-only (storing relationships in SQLite) | SQL is not designed for multi-hop graph traversals; recursive CTEs are awkward |
| NoSQL document store (MongoDB) | Overkill for single-user local-first; adds dependency |
| RDF/SPARQL triple store | Too complex for the problem domain; poor developer UX |
| Embedded Neo4j as default | Adds 100MB+ dependency; overkill for small projects |

## Consequences

### Positive

- Impact analysis, contradiction detection, dependency resolution, and drift tracking are all enabled by the graph model
- Adjacency JSON is zero-dependency, git-friendly, and human-readable
- Neo4j upgrade path exists for projects that need transactional guarantees or Cypher
- NetworkX integration provides Python-native graph algorithms (shortest path, centrality, clustering)

### Negative

- Graph must be kept in sync with `.sspec` files (dual-write requirement)
- Large projects may experience slow JSON parsing for graph files with thousands of nodes
- The adjacency JSON format is not indexed — all traversals are O(n) on edges
